# Learning Model

## Purpose

The project teaches transferable musical relationships on guitar. A learner
should not only know that a note, scale, or chord exists; they should be able to
hear it, locate it, name its role, play it, predict what may follow, and move the
same relationship to another root or neck position.

## The Core Reference Chain

Every musical fact needs a reference:

```text
tonal centre
  -> scale degree
  -> chord root and chord tone
  -> harmonic function
  -> fretboard interval shape
  -> sound and physical action
```

These relationships must be connected without being conflated:

- A note name identifies pitch spelling in context.
- An interval describes distance from a stated root.
- A scale degree describes a note relative to a tonal centre.
- A chord tone describes a note relative to a chord root.
- A Roman numeral describes a chord relative to a key or mode.
- Harmonic function describes contextual behaviour, not chord quality alone.
- A fretboard shape is a physical arrangement of intervals, not merely a diagram name.

## The Learning Loop

Guided activities should follow this pattern:

```text
predict -> select or play -> hear -> receive feedback -> explain -> retry -> transfer
```

Reveal is useful after an attempt. Showing every answer by default turns a
learning activity into passive browsing.

## Representation Contract

Important ideas should connect four representations:

```text
sound <-> interval or function <-> note name <-> fret location and hand action
```

For example, changing a major third to a minor third should be heard, labelled
`3 -> b3`, shown as a one-fret movement, and applied inside a playable shape.

## Progression of Difficulty

1. Establish a tonal centre and root landmarks.
2. Work in a small, playable fretboard region.
3. Build a scale from intervals.
4. Extract chord tones from the scale.
5. Compare one controlled musical change.
6. Retrieve the relationship with reduced labels.
7. Apply it over sound.
8. Transfer the same structure to a new root, shape, or progression.

Advanced harmony, full-neck views, alternate systems, and dense controls should
appear only when prerequisites make them meaningful.

## Music-Theory Contract

- Derive labels from shared theory models wherever practical.
- Preserve musically correct spelling in the active key or mode.
- Do not label harmonic-minor behaviour as natural minor.
- Keep chord intervals and scale degrees visually distinct.
- Explain borrowed, modal, blues, and secondary-dominant behaviour in context.
- When a model is intentionally limited, say what it recognises and what it does not.

## Guitar Contract

- Mark roots consistently and make string/fret orientation unambiguous.
- Teach octave geography, triads, inversions, compact voicings, and voice movement.
- Describe CAGED forms through roots and intervals rather than shape names alone.
- Prefer a playable local map to an undifferentiated full-neck display.
- Connect every theory explanation to a sound or guitar action when possible.

## Evidence of Learning

Progress should represent observable capability, such as finding a requested
root, building a scale or triad, identifying a heard relationship, choosing a
chord tone, or playing a shape. Click counts and elapsed time are not sufficient
evidence by themselves.

The current app records attempts, correctness, streak, context, due date, and a
simple expanding review interval. This is useful scheduling evidence, not a
claim of complete musical mastery.
